# DECISIONS.md — Architecture Decision Records (ADR Log)

> Log of key architecture decisions. Read this to understand WHY things are built the way they are.

## ADR-001: Fastify over NestJS for API

**Date:** Pre-baseline
**Status:** Accepted
**Context:** Need a TypeScript API framework. NestJS is popular but heavy. Fastify is schema-first and 2x Express throughput.
**Decision:** Use Fastify with direct route registration in `apps/api/src/main.ts`.
**Consequences:** Single-file route registration (currently 1,808 lines). May need to split into route modules as we grow. No decorators, no DI container — simpler but requires discipline.

---

## ADR-002: BullMQ over Temporal/Inngest for Job Queue

**Date:** Pre-baseline
**Status:** Accepted
**Context:** Need background job processing for scans, imports, enrichment, notifications. Options: BullMQ (Redis-backed, TS-native), Temporal (heavy, Java roots), Inngest (cloud-first).
**Decision:** BullMQ with Redis 7.
**Consequences:** Simple, proven, TypeScript-native. Cron scheduling built-in. Trade-off: no built-in workflow orchestration (job chains are manual). Acceptable for our job complexity.

## ADR-003: Dual-Layer Multi-Tenancy (Prisma Middleware + PostgreSQL RLS)

**Date:** v3 Spec
**Status:** Accepted
**Context:** Multi-tenant platform where a missed WHERE clause leaks client data. Single-layer protection is insufficient when AI coding agents write queries.
**Decision:** Prisma client extension auto-injects `projectId` (Layer 1) + PostgreSQL RLS policies on all tenant-scoped tables (Layer 2).
**Consequences:** More complex setup. Requires raw SQL migrations for RLS. `SET LOCAL app.current_project_id` must be called per-transaction. But cross-tenant leakage is prevented even if application code has bugs.

## ADR-004: OpenRouter as Primary LLM Gateway

**Date:** v3 Spec
**Status:** Accepted
**Context:** Need AI features (remediation guidance, NL search, asset classification). Options: direct Claude API, direct OpenAI API, self-hosted Ollama, OpenRouter.
**Decision:** OpenRouter API via `openai` npm SDK with configurable `LLM_BASE_URL`.
**Consequences:** Single API key accesses 200+ models. No Docker container needed. Switch providers by changing env vars — no code changes. Trade-off: depends on OpenRouter availability. Mitigation: fallback to NVD advisory link when LLM unavailable.

## ADR-005: MinIO over Cloud S3 for Object Storage

**Date:** v3 Spec
**Status:** Accepted
**Context:** Need object storage for evidence files, report PDFs, export artifacts. Options: AWS S3, Cloudflare R2, MinIO (self-hosted).
**Decision:** MinIO self-hosted as default. S3-compatible API via `@aws-sdk/client-s3`.
**Consequences:** $0/month cost. Data stays on-premises (important for regulated MSP clients). Multi-arch Docker image works on Mac Mini M1. Trade-off: no built-in geo-redundancy. Mitigation: optional cloud S3 via env var swap.

## ADR-006: Zod as Mandatory Validation Layer

**Date:** v3 Spec
**Status:** Accepted
**Context:** API currently uses manual validation. External data (EPSS, KEV, NVD) needs runtime type checking. Env vars should fail fast on startup.
**Decision:** Zod is MANDATORY for all API request/response bodies, external data parsing (`.safeParse()`), environment variable validation, and webhook payloads.
**Consequences:** More upfront schema work. But catches errors at boundaries instead of deep in business logic. `fastify-type-provider-zod` provides automatic OpenAPI schema generation.

## ADR-007: Tailwind + shadcn/ui for UI Stack

**Date:** v3 Spec
**Status:** Accepted
**Context:** Current UI uses plain inline styles. Need a modern component library for consistent design. Options: Material UI, Ant Design, shadcn/ui + Tailwind.
**Decision:** Tailwind CSS + shadcn/ui (Radix primitives) + Lucide React icons + Recharts.
**Consequences:** No CSS modules, no styled-components — Tailwind only. shadcn components are copy-pasted (not npm installed), giving full ownership. Dark mode via Tailwind `dark:` variants + `next-themes`. Trade-off: existing 10 pages need migration from inline styles (INFRA-10).

## ADR-008: EPSS over CVSS as Primary Risk Indicator

**Date:** v3 Spec
**Status:** Accepted
**Context:** CVSS scores are theoretical severity ratings. EPSS scores predict actual exploitation probability. Research shows EPSS is a better predictor of real-world attacks.
**Decision:** Composite score weights EPSS at 0.30 (highest), CVSS at 0.25, KEV at 0.20, blast radius at 0.15, exposure at 0.10.
**Consequences:** Vulns with high CVSS but low EPSS rank lower. Vulns with low CVSS but high EPSS + KEV status rank higher. This matches real-world threat landscape better than CVSS-only sorting.

## ADR-009: Composite Score Formula

**Date:** v3 Spec
**Status:** Accepted
**Context:** Need a single prioritisation number combining multiple risk signals.
**Decision:**
```
compositeScore = (
  (cvssNormalised * 0.25) +
  (epssNormalised * 0.30) +
  (kevBonus * 0.20) +
  (blastRadiusNorm * 0.15) +
  (exposureBonus * 0.10)
) / totalWeight
```
Weights configurable per project. Output: 0–100 score.
**Consequences:** Deterministic, explainable score. Breakdown visible in UI tooltip. Requires EPSS and KEV sync to be operational (daily cron jobs).

## ADR-010: Single Prisma Schema File

**Date:** Pre-baseline
**Status:** Accepted
**Context:** Could split Prisma schema across multiple files (using prisma-merge or similar). Opted for single file.
**Decision:** `apps/api/prisma/schema.prisma` is the SINGLE source of truth for the data model.
**Consequences:** File will grow large (20+ models). But: single grep target, no merge conflicts, clear ownership. All models use `@@map("snake_case_table_name")`.

## ADR-011: Phase 9 Agents in Go (not TypeScript)

**Date:** v3 Spec
**Status:** Accepted
**Context:** Host telemetry agents need to run on Linux, Windows, macOS as lightweight daemons. Options: TypeScript (Node.js), Rust, Go.
**Decision:** Go for agent binaries. Cross-compile to 6 targets (linux/windows/darwin × amd64/arm64).
**Consequences:** Separate language from main stack. But: single binary deployment (no runtime), <50MB memory, <2% CPU. mTLS for authentication. eBPF for Linux process monitoring, ETW for Windows.

## ADR-012: Signed Sessions over JWT for Web Auth

**Date:** Pre-baseline
**Status:** Accepted
**Context:** JWT is stateless but can't be revoked easily. Signed sessions store state server-side.
**Decision:** Signed session cookies for web auth. API keys (SHA-256 hashed) for integrations.
**Consequences:** Session state in PostgreSQL. Immediate revocation possible. Dev mode uses header trust for convenience (`AUTH_ALLOW_LEGACY_HEADERS=true`).

## ADR-013: Essential Eight as Primary Compliance Framework

**Date:** v3 Spec
**Status:** Accepted
**Context:** Target market is Australian MSPs. ASD Essential Eight is the dominant compliance framework. Also need CIS Controls and NIST CSF support.
**Decision:** Essential Eight as primary (P0), with multi-framework support (CIS, NIST) as P1.
**Consequences:** ComplianceFramework + ComplianceControl + ComplianceMapping models support any framework. Essential Eight mapping is seeded first. Auto-assessment from scan data where possible.

---

## ADR-014: Monorepo with Turborepo over Nx

**Date:** Pre-baseline
**Status:** Accepted
**Context:** Need monorepo tooling for 3 apps + shared packages. Options: Turborepo, Nx, Rush.
**Decision:** Turborepo with pnpm workspaces.
**Consequences:** Simpler than Nx. Parallel builds with caching. Trade-off: less sophisticated dependency graph analysis. Acceptable for our repo size.

---

## ADR-015: ConnectWise + HaloPSA as Launch PSA Integrations

**Date:** v3 Spec
**Status:** Accepted
**Context:** MSP market uses various PSA tools. Need to pick initial targets. ConnectWise Manage has ~40% MSP market share. HaloPSA is growing rapidly in the AU/UK market.
**Decision:** ConnectWise Manage (REST API, npm package `connectwise-rest`) and HaloPSA (REST API, OAuth2) as P0 integrations. Polymorphic PSA connector pattern for easy addition of others.
**Consequences:** Two integrations to build and maintain at launch. Polymorphic design means adding Autotask, Syncro, etc. later is incremental work.
