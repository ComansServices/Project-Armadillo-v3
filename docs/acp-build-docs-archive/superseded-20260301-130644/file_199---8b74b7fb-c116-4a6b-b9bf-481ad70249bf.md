## System Overview

Armadillo v3 is a queue-driven network discovery and security visibility platform for small-to-mid MSPs. It runs as a self-hosted Docker Compose stack with optional Kubernetes scale-out.

## Data Flow

### Request Pipeline

```
Browser → Next.js (SSR/CSR) → Fastify API → PostgreSQL (via Prisma)
                                    │
                                    ├── BullMQ Job Queue → Redis
                                    │         │
                                    │         └── Worker processes job
                                    │               ├── Scan execution
                                    │               ├── XML import parsing
                                    │               ├── EPSS/KEV sync
                                    │               ├── Enrichment pipeline
                                    │               ├── Report generation
                                    │               ├── AI guidance (→ OpenRouter)
                                    │               ├── PSA ticket creation
                                    │               └── Notification delivery
                                    │
                                    └── MinIO (evidence, reports)
```

### Daily Enrichment Chain (Scheduled)

```
02:00 UTC  sync-epss-scores     → Download CSV → Upsert EpssCache
02:30 UTC  sync-cisa-kev        → Download JSON → Upsert CisaKevEntry
           enrich-vulnerabilities → Join EPSS+KEV → Update Vulnerability records
           (triggered after sync)   → Compute compositeScore via packages/scoring
03:00 UTC  compute-posture-scores → Aggregate per project → Insert PostureSnapshot
03:30 UTC  compute-compliance-maps → Map vulns to frameworks → Update ComplianceMapping
07:00 LCL  send-daily-digests    → Generate per-user digest → Route via packages/notifications
```

### Event-Driven Jobs

```
Vuln assigned       → create-psa-ticket (ConnectWise/Halo)
Vuln assigned       → deliver-webhook (registered subscribers)
Scan complete       → generate-scan-digest (AI summary)
User clicks "Fix"   → generate-ai-guidance (OpenRouter, cached 7d)
Asset created       → auto-classify-asset (AI classification, cached 30d)
Evidence uploaded   → store to MinIO → link to EvidenceAttachment
EPSS spike detected → detect-epss-spikes → alert via notification router
```

## Multi-Tenancy Architecture

### Dual-Layer Isolation

Every tenant-scoped query is protected at TWO layers:

**Layer 1 — Prisma Middleware (Application)**
- Prisma client extension auto-injects `where: { projectId }` on every query
- `projectId` read from AsyncLocalStorage (set by auth middleware per request)
- Cannot be bypassed by individual route handlers
- `prisma.$bypass()` escape hatch for owner-level admin queries only

**Layer 2 — PostgreSQL Row-Level Security (Database)**
- RLS policies on ALL tenant-scoped tables
- `SELECT/INSERT/UPDATE/DELETE WHERE projectId = current_setting('app.current_project_id')`
- Context set per-transaction: `SET LOCAL app.current_project_id = $1`
- Even if Prisma middleware fails, the database itself prevents cross-tenant leakage

### Tenant-Scoped Tables (RLS Required)

Asset, Vulnerability (AssetVulnerability), Scan, ScanEvent, ScanSchedule, XmlImport, ImportSourcePolicy, Report, SavedView, WebhookSubscription, ApiKey, ComplianceMapping, EvidenceAttachment, AuditEvent, PostureSnapshot, Runbook, RunbookProgress, ScanCredential, NotificationPreference

### Non-Tenant Tables (No RLS)

User, EpssCache, CisaKevEntry, ComplianceFramework, ComplianceControl, ThreatIntelFeed, ThreatIndicator

## Authentication & RBAC

### Auth Modes

| Mode | When | Mechanism |
|---|---|---|
| Signed Session | Production | Encrypted session cookie |
| Header Trust | Dev only | `x-armadillo-user`, `x-armadillo-role`, `x-armadillo-auth` |
| API Key | Integrations | `Authorization: Bearer armadillo_xxx` (SHA-256 hashed) |

Header trust is disabled in production via `AUTH_ALLOW_LEGACY_HEADERS=false`.

### RBAC Roles

| Role | Scope | Permissions |
|---|---|---|
| **owner** | All projects | Full access. Create/delete projects, manage team, billing. |
| **admin** | Assigned projects | Full access to assigned projects. Manage scans, vulns, reports. |
| **staff** | Assigned projects | Operational access. Triage vulns, run scans, attach evidence. |
| **viewer** | Assigned projects | Read-only. Used for customer portal (US-8.01). |

### Identity Model

```
User (email, displayName, passwordHash, isActive, isMfaEnabled)
  └── ProjectMembership (userId, projectId, role, grantedBy, grantedAt)
        └── Project (name, description, cidrRanges, brandingConfig, postureScore)
```

## Deployment Tiers

### Tier 1 — Single Machine (Default)

Target: MSPs serving 1–20 clients. Mac Mini M1 (16GB RAM) or Linux server.

```yaml
# docker-compose.yml
services:
  postgres:    postgres:16-alpine
  redis:       redis:7-alpine
  minio:       minio/minio:latest      # Ports 9000/9001
  api:         apps/api (Fastify)       # Port 4000
  worker:      apps/worker (BullMQ)     # Concurrency 4
  web:         apps/web (Next.js)       # Port 3000
```

Capacity: 50+ projects, 100+ scans/day, 10K+ vulns on Mac Mini M1 16GB.

### Tier 2 — Multi-Machine (2–4 Servers)

Split: Machine A (PostgreSQL + Redis), Machine B (API + Web), Machine C (Worker), optional Machine D (Scanner).

### Tier 3 — Kubernetes (Optional)

Helm charts, KEDA auto-scaling on BullMQ queue depth, horizontal pod autoscaler for API/Web.

## Platform Compatibility

**HARD REQUIREMENT:** ALL Docker images MUST support `linux/amd64` AND `linux/arm64` natively. No Rosetta emulation.

Verified multi-arch images: postgres:16, redis:7-alpine, minio/minio:latest, node:20-slim.

Armadillo Dockerfiles use `node:20-slim` base. No platform-specific paths or binaries.

macOS dev: Docker Desktop (disable Rosetta) or Colima (`colima start --arch aarch64`).

## Key Architecture Decisions

| Decision | Choice | Rationale |
|---|---|---|
| API framework | Fastify (NOT NestJS) | 2x Express throughput, schema-first, lightweight |
| ORM | Prisma | Type-safe queries, migration management, introspection |
| Queue | BullMQ | Redis-backed, TypeScript-native, cron scheduling built-in |
| Build system | Turborepo + pnpm | Parallel builds, caching, workspace isolation |
| Web framework | Next.js 14 (App Router) | SSR, API routes, React Server Components |
| UI stack | Tailwind + shadcn/ui | Utility-first CSS, Radix primitives, no runtime overhead |
| LLM provider | OpenRouter (OpenAI SDK) | Single key for 200+ models, provider-agnostic via env vars |
| Object storage | MinIO (self-hosted) | S3-compatible, AGPL-3.0, multi-arch, zero cloud cost |
| Validation | Zod (mandatory) | Runtime type safety for API boundaries, env vars, external data |
| Multi-tenancy | Prisma middleware + PostgreSQL RLS | Dual-layer isolation, defence-in-depth |
