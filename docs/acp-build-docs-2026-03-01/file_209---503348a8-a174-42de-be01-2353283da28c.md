## 1. Executive Summary

Project Armadillo v3 is a modern, queue-driven network discovery and
security visibility platform built by Comans Services. It bridges the
gap between enterprise-grade SIEMs (which cost $50K--$500K/year and
require dedicated teams) and fragmented open-source scanners (which
require manual stitching and tribal knowledge).

Armadillo is designed specifically for small-to-mid MSPs with 10--50
employees who need to offer security-as-a-service to their clients
without hiring a dedicated security team. It provides
exploitability-first vulnerability prioritisation, attack path
simulation, remediation workflow tracking, automated reporting, and
multi-tenant project scoping.

The platform is self-hosted via Docker (5-minute deploy), free at the
core with support tiers, and designed to reduce manual report assembly
from 4 hours to 10 minutes while enabling junior staff to run security
operations without constant senior oversight.

This specification contains 63 user stories across 9 sections, covering
the immediate sprint backlog, three major roadmap phases, and six
cross-cutting feature themes that position Armadillo as a competitive,
market-ready MSP security platform. The document includes a complete
Prisma data model with 20+ entities, PostgreSQL Row-Level Security for
multi-tenant isolation, formal RBAC with User and ProjectMembership
models, encrypted scan credential vault, and mandatory Zod validation
across all API boundaries.

### 1.1 Target Users

-   MSP owners who want to offer security-as-a-service without building
    a SOC

-   IT managers proving security posture to auditors and boards

-   Security analysts tired of spreadsheet-driven vulnerability
    management

-   Compliance officers needing evidence of continuous monitoring

### 1.2 Core Value Propositions

-   Exploitability-first prioritisation: Fix 20% of vulns that prevent
    80% of breaches

-   Attack path simulation: Understand actual risk, not just theoretical
    CVSS scores

-   Remediation tracking: Measure time-to-fix, hold staff accountable

-   Automated reporting: Reduce assembly from 4 hours to 10 minutes

-   Change awareness: Spot new assets and drift before they become
    shadow IT

-   Quality-controlled imports: Bad data does not pollute security
    decisions

### 2.4 Authentication & Multi-Tenancy Model

API authentication uses signed session auth in production. Development
uses header-based auth for convenience:

x-armadillo-user: <actor>

x-armadillo-role: owner \| admin \| staff \| viewer

Legacy header trust is disabled in production. Project scope enforcement
is active for scan and schedule paths. Auth audit and lockout controls
are active.

### 2.4.1 Multi-Tenancy Isolation Strategy

Armadillo is a multi-tenant platform where a single database serves
multiple MSP clients. A missed projectId filter in any API route would
leak one client's vulnerabilities to another. To prevent this
catastrophic failure mode, Armadillo enforces tenant isolation at TWO
layers:

6.  Application Layer (Prisma Middleware): A Prisma client extension
    automatically injects where: { projectId } into every query. The
    active projectId is set from the authenticated session context at
    the start of every API request. This middleware cannot be bypassed
    by individual route handlers.

7.  Database Layer (PostgreSQL Row-Level Security): RLS policies on all
    tenant-scoped tables (Vulnerability, Asset, Scan, Import, Schedule,
    Report, etc.) enforce that rows are only visible when
    app.current_project_id matches the row's projectId. The API sets
    this context per-transaction using SET LOCAL app.current_project_id
    = $1.

The dual-layer approach means that even if an AI coding agent or junior
developer writes a query that forgets to filter by projectId, the
database itself will prevent cross-tenant data leakage. RLS policies are
defined in the Prisma migration as raw SQL.

### 2.4.2 User Identity & RBAC

The system uses a formal User model with ProjectMembership to answer the
question: which projects can this user access? Roles are defined as:

-   owner: Full access to all projects. Can create/delete projects,
    manage team, billing.

-   admin: Full access to assigned projects. Can manage scans, vulns,
    reports, schedules.

-   staff: Operational access to assigned projects. Can triage vulns,
    run scans, attach evidence.

-   viewer: Read-only access to assigned projects. Used for customer
    portal (US-8.01).

See Section 18.4 for the complete User, ProjectMembership, and Role
schema.

### 2.4 Data Flow

The system follows a queue-driven pipeline architecture:

8.  User uploads XML scan data (Nessus/OpenVAS) or queues a new scan via
    the API

9.  API validates input and enqueues a BullMQ job

10. Worker picks up the job, processes scan data with quality controls
    (strict/lenient modes)

11. Parsed assets, vulnerabilities, and services are persisted to
    PostgreSQL

12. Web UI reads from API to display dashboards, vulnerability lists,
    asset inventories, and reports

13. Scheduled scans are dispatched automatically via cron expressions
    with conflict detection

## 6. Data Model (Inferred from API)

The following data model is inferred from the API endpoints and UI
features documented in the README. The actual Prisma schema in
apps/api/prisma is the source of truth.

### 6.1 Core Entities

-   Project: Multi-tenant container. All scans, assets, vulns scoped to
    a project.

-   Scan: Represents a scan job. Fields: projectId, requestedBy, targets
    (CIDR/IP), config (profile), status
    (queued/running/completed/failed), timestamps.

-   Asset: Discovered host. Fields: IP, hostname, OS, ports, services,
    projectId, deltaSinceLast (change tracking), risk score, badge
    (new/new_this_week/changed).

-   Vulnerability: CVE/finding. Fields: cveId, severity, description,
    hasExploit (boolean), blastRadius (affected host count), assignedTo,
    dueDate, remediationStatus
    (open/in_progress/resolved/risk_accepted/false_positive),
    statusReason (required for risk_accepted/false_positive), assetId,
    projectId.

-   Import: XML import record. Fields: source, qualityMode
    (strict/lenient), status, quality analytics (success rate, skip
    reasons, invalid count), reject artifacts.

-   Schedule: Recurring scan definition. Fields: projectId,
    cronExpression, targets, paused (boolean), pauseReason, conflict
    detection data.

-   Report: Generated report. Fields: type (PDF/CSV), deliveryStatus
    (sent/pending/acknowledged), archive path, failure notifications.

CRITICAL: TelemetryEvent will generate millions of rows per day across a
fleet. The following data management strategy is MANDATORY:

14. PostgreSQL table partitioning: Partition TelemetryEvent by RANGE on
    timestamp using weekly partitions. Implement via raw SQL in Prisma
    migration (CREATE TABLE telemetry_event PARTITION BY RANGE
    (timestamp)).

15. Retention policy: Default retention is 30 days for info-severity
    events, 90 days for warning, and 365 days for critical. Configurable
    per-project.

16. Pruning cron: The prune-telemetry BullMQ job (04:00 UTC daily) drops
    partitions older than the retention window using DROP TABLE IF
    EXISTS telemetry_event_YYYYWW. This is O(1) regardless of data
    volume.

17. Aggregation before deletion: Before dropping a partition, the prune
    job rolls up daily summaries (event counts by type, top processes,
    top network destinations) into a TelemetrySummary table for
    long-term trend analysis.

### 19.1 FIRST.org EPSS API

  **Property**       **Details**

  Full CSV Download  https://epss.empiricalsecurity.com/epss_scores-YYYY-MM-DD.csv.gz

## 20. Infrastructure Architecture

### 20.1 Platform Compatibility

Armadillo MUST run on both linux/amd64 and linux/arm64 architectures.
This is a hard requirement because the target deployment platforms
include x86 Linux servers, Apple Silicon Mac Minis (M1/M2/M4), and mixed
clusters. ALL Docker images used in the stack must provide multi-arch
manifests for both platforms. No image may require –platform
linux/amd64 emulation in production.

Verified ARM64-native Docker images (all confirmed multi-arch):

-   PostgreSQL (postgres:16): Official multi-arch — amd64, arm64v8. No
    issues on Apple Silicon.

-   Redis (redis:7-alpine): Official multi-arch — amd64, arm64v8.
    Native performance on ARM.

-   MinIO (minio/minio:latest): Official multi-arch — amd64, arm64.
    Confirmed working on M1.

-   Node.js (node:20-slim): Official multi-arch — amd64, arm64. Used
    as base image for API, Worker, and Web.

The Armadillo Dockerfiles in apps/api/Dockerfile,
apps/worker/Dockerfile, and apps/web/Dockerfile MUST use the multi-arch
node:20-slim base image and must NOT hardcode any platform-specific
paths or binaries.

### 20.2 Deployment Tiers

Armadillo supports three deployment tiers, from single-machine to
clustered. The Docker Compose configuration is the primary deployment
method. Kubernetes is optional for scale-out.

**Tier 1 — Single Machine (Mac Mini M1 or Linux Box)**

The default and recommended production deployment for MSPs serving 1--20
clients. Runs the entire stack on a single Mac Mini M1 (16GB RAM) or
equivalent Linux server. Docker Compose orchestrates all containers:
API, Worker, Web, PostgreSQL, Redis, MinIO. This is the 5-minute deploy
target.

-   Hardware: Mac Mini M1/M2/M4 (16GB+ RAM, 256GB+ SSD) or Linux server
    (Ubuntu 22.04+, 16GB+ RAM)

-   macOS Docker: Docker Desktop for Mac (recommended) or Colima. Both
    support ARM64 natively.

-   Linux Docker: Docker Engine 24+ with Docker Compose v2. No emulation
    needed on ARM64 Linux (e.g., Raspberry Pi 5, Ampere, Graviton).

-   Capacity: 50+ projects, 100+ scheduled scans/day, 10,000+
    vulnerabilities. Tested on Mac Mini M1 16GB.

-   Deployment: make up (single command). Makefile handles docker
    compose up, Prisma migrations, and seed data.

**Tier 2 — Multi-Machine (2--4 Mac Minis or Linux Servers)**

For MSPs serving 20--50+ clients or needing scan isolation. Split the
stack across machines:

-   Machine A: PostgreSQL + Redis (data tier). Persistent storage,
    backed up.

-   Machine B: API + Web + MinIO (application tier). Stateless, easy to
    replace.

-   Machine C: Worker(s) (scan processing tier). Multiple workers can
    share a single BullMQ queue. Add machines here to scale scan
    throughput.

-   Machine D (optional): OpenVAS (open source, GPL) or Nessus
    (proprietary, optional) scanner. Dedicated scan engine with
    credentials stored in Armadillo. Armadillo imports XML scan results
    from either scanner — the scan engine is external and
    interchangeable.

Machines communicate over a private LAN. PostgreSQL and Redis accept
connections from the other machines via standard TCP. No Kubernetes
required.

**Tier 3 — Kubernetes (Optional Scale-Out)**

For large MSPs or SaaS deployments serving 50+ clients with elastic
scaling requirements. Use Kubernetes on any platform that supports it:

-   Self-hosted: k3s on Linux/Mac (lightweight, ARM64-native, single
    binary). Recommended for small-team Kubernetes.

-   Cloud: AWS EKS (Graviton ARM64 instances for cost savings), GKE,
    Azure AKS, or DigitalOcean Kubernetes.

-   Auto-scaling: KEDA scales Worker pods based on BullMQ queue depth.

-   Database: Managed PostgreSQL (RDS, Cloud SQL, Neon) or self-hosted
    with pgBackRest for backup.

Helm charts and manifests are in k8s/ directory. All container images
are built as multi-arch (linux/amd64 + linux/arm64) via GitHub Actions
using docker buildx.

### 20.3 Infrastructure Components

The production architecture adds five new infrastructure components to
the existing stack of Fastify API, Next.js Web, BullMQ Worker,
PostgreSQL, and Redis:

18. Object Storage (MinIO — self-hosted default, S3-compatible):
    Stores evidence attachments (screenshots, logs), report PDFs, and
    export artifacts. Referenced by s3Key fields in EvidenceAttachment
    and Report models. MinIO is open source (AGPL-3.0), S3-API
    compatible, and runs on ARM64. Any S3-compatible service works as a
    drop-in replacement.

19. LLM Gateway: A thin proxy service within apps/api that routes AI
    requests to OpenRouter (default) or any OpenAI-compatible endpoint
    with caching, rate limiting, and cost tracking. Provider is
    configured via LLM_BASE_URL and LLM_API_KEY env vars. Default:
    OpenRouter API (https://openrouter.ai/api/v1).

20. Notification Router: A BullMQ job processor that routes alerts to
    configured channels (email via SMTP/nodemailer, Teams webhooks,
    Slack webhooks, in-app via WebSocket). Reads NotificationPreference
    to determine routing.

21. Intelligence Sync Service: A set of scheduled BullMQ jobs (cron)
    that fetch EPSS scores, CISA KEV data, and threat intel feeds daily,
    then trigger enrichment jobs to update vulnerability records.

22. Telemetry Ingestion Service (Phase 9): A dedicated Fastify endpoint
    (/api/v1/telemetry/batch) that receives agent telemetry via mTLS and
    writes to PostgreSQL with partitioned tables for high-volume event
    data.

### 20.3 Composite Risk Score Formula

The composite score (US-INT.03) is the single most important new
computation. It must be deterministic, explainable, and configurable.
Recommended formula in packages/scoring:

compositeScore = (

(cvssNormalised * weights.cvss) + // CVSS 0-10 mapped to 0-100

(epssNormalised * weights.epss) + // EPSS 0-1 mapped to 0-100

(kevBonus * weights.kev) + // 100 if in KEV, else 0

(blastRadiusNorm * weights.blast) + // log10(affectedHosts) * 25

(exposureBonus * weights.exposure) // 100 if internet-facing, else 0

) / totalWeight

Default weights: cvss=0.25, epss=0.30, kev=0.20, blast=0.15,
exposure=0.10. These weights are stored in the project settings and can
be adjusted per deployment. The EPSS weight is highest by default
because research shows it is the best single predictor of real-world
exploitation.

## 23. Non-Functional Requirements

### 23.1 Performance

-   Dashboard pages must load in under 2 seconds with 1,000+ assets

-   Vulnerability list must paginate smoothly with 10,000+ records

-   Attack path simulation must return results in under 5 seconds for
    networks up to 500 nodes

-   XML import of a 50MB Nessus scan must complete within 60 seconds

-   Global search must return results in under 500ms

-   EPSS/KEV enrichment must not add more than 200ms to vulnerability
    list load time

-   AI remediation guidance must return within 5 seconds

### 23.2 Scalability

-   Tier 1 (single Mac Mini M1 16GB): Support 50+ concurrent projects,
    100+ scheduled scans/day, 10,000+ vulnerabilities

-   Tier 2 (multi-machine): Horizontal worker scaling via BullMQ shared
    queue across machines on same LAN

-   Tier 3 (Kubernetes): Auto-scale workers based on queue depth (KEDA).
    ARM64-native k3s or cloud Kubernetes.

-   Phase 9 agents: support 5,000+ endpoints reporting telemetry

-   EPSS and KEV data must be cached and refreshed daily without
    impacting API performance

### 23.3 Security

-   Multi-tenancy: PostgreSQL Row-Level Security (RLS) policies MUST be
    active on ALL tenant-scoped tables. RLS is the last line of defence
    against cross-tenant data leakage.

-   Multi-tenancy: Prisma middleware MUST inject projectId filtering on
    every query. The API MUST call SET LOCAL app.current_project_id at
    the start of every transaction.

-   Multi-tenancy: Integration tests MUST verify that User A in Project
    X cannot read data from Project Y, even with direct Prisma queries
    bypassing the API.

-   RBAC: All API endpoints must check ProjectMembership before allowing
    access. A user with no membership for a projectId receives 403, not
    an empty result.

-   RBAC: The viewer role (customer portal) must have read-only access
    enforced at both API and database levels.

-   Credentials: Scan credentials (ScanCredential model) must be
    encrypted at rest using AES-256-GCM. Decrypted payloads must NEVER
    be returned via the API and ONLY passed to the worker at scan
    execution time.

## 25. Technical Constraints & Decisions

### 25.1 Architecture Decisions

-   Queue-driven: BullMQ chosen for Redis backing, built-in retry logic,
    and TypeScript-native support

-   Fastify over Express: chosen for performance (2x throughput) and
    schema-first validation

-   Prisma ORM: chosen for type-safe database access, migration
    management, and developer ergonomics

-   Turborepo: chosen for build caching and parallel task execution in
    the monorepo

-   Next.js: chosen for the web UI to leverage SSR, API routes, and the
    React ecosystem

-   PostgreSQL RLS: Row-Level Security policies on all tenant-scoped
    tables as the database-level safety net for multi-tenancy isolation,
    supplementing application-level Prisma middleware

### 25.2 UI Component Stack (MANDATORY for AI Agents)

AI coding agents MUST use the following UI stack for all interface
components. This prevents inconsistent styling choices and hallucinated
CSS modules:

-   Styling: Tailwind CSS for all utility-class-based styling. No raw
    CSS modules, no styled-components, no inline style objects.

-   Components: shadcn/ui as the component library (built on Radix UI
    primitives). Use shadcn components for buttons, dialogs, tables,
    forms, dropdowns, toasts, and navigation.

-   Icons: Lucide React for all iconography. No mixing icon libraries.

-   Charts: Recharts for dashboard visualisations (line, bar, area,
    pie). Consistent across all views.

-   Forms: React Hook Form with Zod resolvers for all form validation.

-   Dark mode: Use Tailwind dark: variants with next-themes for theme
    toggling (US-OPS.04).

### 25.3 Schema Validation (MANDATORY)

AI coding agents MUST use Zod for ALL of the following. This is
non-negotiable — AI agents are unreliable at handling edge-case nulls
without strict schemas:

-   Fastify API request body validation: Every POST/PATCH endpoint must
    define a Zod schema. Use fastify-type-provider-zod for Fastify
    integration.

-   Fastify API response validation: Response schemas ensure the API
    contract is enforced even when Prisma returns unexpected shapes.

-   External API response parsing: EPSS CSV rows, CISA KEV JSON, NVD API
    responses, PSA/RMM API responses must all be parsed through Zod
    schemas with .safeParse() to handle malformed data gracefully.

-   Environment variable validation: Use Zod to validate all env vars at
    startup (apps/api/src/config.ts). Fail fast if required vars are
    missing.

-   Webhook payload validation: Both incoming (from PSA callbacks) and
    outgoing (to subscriber endpoints) webhook payloads must conform to
    versioned Zod schemas.

### 25.4 Constraints for Agents

-   Phase 9 agents MUST be written in Go (cross-compilation, small
    binary, no runtime dependency)

-   Go agents MUST be compiled for all six targets: linux/amd64,
    linux/arm64, darwin/amd64, darwin/arm64, windows/amd64,
    windows/arm64. Use GOOS and GOARCH environment variables in the CI
    build.

-   Agent to server communication MUST use mTLS (mutual TLS) with
    certificate pinning

-   eBPF is Linux-only; Windows uses ETW; macOS uses ESF. Each platform
    has a separate collection strategy

-   Agents must have a maximum memory footprint of 50MB and CPU usage
    under 2% average

### 25.5 Intelligence Enrichment Constraints

-   EPSS scores must be fetched daily from FIRST.org API and cached
    locally in PostgreSQL

-   CISA KEV catalog must be synced daily via the official JSON feed

-   Composite risk score formula must be configurable (weights
    adjustable per deployment)

-   External API failures must not block core vulnerability display
    (graceful degradation)

### 25.6 Integration Constraints

-   PSA integrations must use official REST APIs (no screen scraping or
    unofficial endpoints)

-   Webhook payloads must follow a documented JSON schema with
    versioning

-   SIEM forwarding must support both CEF and raw JSON over syslog
    (TCP/UDP)

-   All integrations must be configurable per-project (not global-only)

### 25.7 Repository Conventions

-   All code is TypeScript (strict mode) unless platform-specific (Go
    for agents, Python for scripts)

-   Prisma migrations are the source of truth for the database schema

-   Docker Compose is the primary deployment method (Tier 1 and Tier 2).
    Kubernetes manifests in k8s/ are optional for Tier 3.

-   All Dockerfiles MUST use multi-arch base images (node:20-slim, not
    node:20-slim-amd64). CI builds via docker buildx for linux/amd64 +
    linux/arm64.

-   Environment configuration via .env files (not committed) and Docker
    environment variables

-   Dev mode: ARMADILLO_DEV_MODE=1 suppresses alert notifications

-   macOS development: Docker Desktop (recommended) or Colima with
    –arch aarch64. Rosetta emulation must NOT be required.
