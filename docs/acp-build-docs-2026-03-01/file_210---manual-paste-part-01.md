A missed projectId filter in any API route would leak one client’s vulnerabilities to another. To prevent this catastrophic failure mode, Armadillo enforces tenant isolation at TWO layers: 1. Application Layer (Prisma Middleware): A Prisma client extension automatically injects where: { projectId } into every query. The active projectId is set from the authenticated session context at the start of every API request. This middleware cannot be bypassed by individual route handlers. 1. Database Layer (PostgreSQL Row-Level Security): RLS policies on all tenant-scoped tables (Vulnerability, Asset, Scan, Import, Schedule, Report, etc.) enforce that rows are only visible when app.current_project_id matches the row’s projectId. The API sets this context per-transaction using SET LOCAL app.current_project_id = $1. The dual-layer approach means that even if an AI coding agent or junior developer writes a query that forgets to filter by projectId, the database itself will prevent cross-tenant data leakage. RLS policies are defined in the Prisma migration as raw SQL. ### 2.4.2 User Identity & RBAC The system uses a formal User model with ProjectMembership to answer the question: which projects can this user access? Roles are defined as:- owner: Full access to all projects. Can create/delete projects, manage team, billing. - admin: Full access to assigned projects. Can manage scans, vulns, reports, schedules. - staff: Operational access to assigned projects. Can triage vulns, run scans, attach evidence. - viewer: Read-only access to assigned projects. Used for customer portal (US-8.01). See Section 18.4 for the complete User, ProjectMembership, and Role schema. ### 2.4 Data Flow The system follows a queue-driven pipeline architecture: 1. User uploads XML scan data (Nessus/OpenVAS) or queues a new scan via the API 1. API validates input and enqueues a BullMQ job 1. Worker picks up the job, processes scan data with quality controls (strict/lenient modes) 1. Parsed assets, vulnerabilities, and services are persisted to PostgreSQL 1. Web UI reads from API to display dashboards, vulnerability lists, asset inventories, and reports 1. Scheduled scans are dispatched automatically via cron expressions with conflict detection ## 3.
