## 1. Executive Summary
Project Armadillo v3 is a modern, queue-driven network discovery and security visibility platform built by Comans Services. It bridges the gap between enterprise-grade SIEMs (which cost $50K–$500K/year and require dedicated teams) and fragmented open-source scanners (which require manual stitching and tribal knowledge).

Armadillo is designed specifically for small-to-mid MSPs with 10–50 employees who need to offer security-as-a-service to their clients without hiring a dedicated security team. It provides exploitability-first vulnerability prioritisation, attack path simulation, remediation workflow tracking, automated reporting, and multi-tenant project scoping.

The platform is self-hosted via Docker (5-minute deploy), free at the core with support tiers, and designed to reduce manual report assembly from 4 hours to 10 minutes while enabling junior staff to run security operations without constant senior oversight.

This specification contains 63 user stories across 9 sections, covering the immediate sprint backlog, three major roadmap phases, and six cross-cutting feature themes that position Armadillo as a competitive, market-ready MSP security platform. The document includes a complete Prisma data model with 20+ entities, PostgreSQL Row-Level Security for multi-tenant isolation, formal RBAC with User and ProjectMembership models, encrypted scan credential vault, and mandatory Zod validation across all API boundaries.

Infrastructure Architecture ### 20.1 Platform Compatibility Armadillo MUST run on both linux/amd64 and linux/arm64 architectures. This is a hard requirement because the target deployment platforms include x86 Linux servers, Apple Silicon Mac Minis (M1/M2/M4), and mixed clusters. ALL Docker images used in the stack must provide multi-arch manifests for both platforms. No image may require –platform linux/amd64 emulation in production. Verified ARM64-native Docker images (all confirmed multi-arch): - PostgreSQL (postgres:16): Official multi-arch — amd64, arm64v8. No issues on Apple Silicon. - Redis (redis:7-alpine): Official multi-arch — amd64, arm64v8. Native performance on ARM. - MinIO (minio/minio:latest): Official multi-arch — amd64, arm64. Confirmed working on M1. - Node.js (node:20-slim): Official multi-arch — amd64, arm64. Used as base image for API, Worker, and Web. The Armadillo Dockerfiles in apps/api/Dockerfile, apps/worker/Dockerfile, and apps/web/Dockerfile MUST use the multi-arch node:20-slim base image and must NOT hardcode any platform-specific paths or binaries. ### 20.2 Deployment Tiers Armadillo supports three deployment tiers, from single-machine to clustered. The Docker Compose configuration is the primary deployment method. Kubernetes is optional for scale-out. Tier 1 — Single Machine (Mac Mini M1 or Linux Box) The default and recommended production deployment for MSPs serving 1–20 clients. Runs the entire stack on a single Mac Mini M1 (16GB RAM) or equivalent Linux server. Docker Compose orchestrates all containers: API, Worker, Web, PostgreSQL, Redis, MinIO. This is the 5-minute deploy target. - Hardware: Mac Mini M1/M2/M4 (16GB+ RAM, 256GB+ SSD) or Linux server (Ubuntu 22.04+, 16GB+ RAM) - macOS Docker: Docker Desktop for Mac (recommended) or Colima. Both support ARM64 natively. - Linux Docker: Docker Engine 24+ with Docker Compose v2. No emulation needed on ARM64 Linux (e.g., Raspberry Pi 5, Ampere, Graviton). - Capacity: 50+ projects, 100+ scheduled scans/day, 10,000+ vulnerabilities. Tested on Mac Mini M1 16GB. - Deployment: make up (single command). Makefile handles docker compose up, Prisma migrations, and seed data.
