## 1. Executive Summary
Project Armadillo v3 is a modern, queue-driven network discovery and security visibility platform built by Comans Services. It bridges the gap between enterprise-grade SIEMs (which cost $50K–$500K/year and require dedicated teams) and fragmented open-source scanners (which require manual stitching and tribal knowledge).

Armadillo is designed specifically for small-to-mid MSPs with 10–50 employees who need to offer security-as-a-service to their clients without hiring a dedicated security team. It provides exploitability-first vulnerability prioritisation, attack path simulation, remediation workflow tracking, automated reporting, and multi-tenant project scoping.

The platform is self-hosted via Docker (5-minute deploy), free at the core with support tiers, and designed to reduce manual report assembly from 4 hours to 10 minutes while enabling junior staff to run security operations without constant senior oversight.

This specification contains 63 user stories across 9 sections, covering the immediate sprint backlog, three major roadmap phases, and six cross-cutting feature themes that position Armadillo as a competitive, market-ready MSP security platform. The document includes a complete Prisma data model with 20+ entities, PostgreSQL Row-Level Security for multi-tenant isolation, formal RBAC with User and ProjectMembership models, encrypted scan credential vault, and mandatory Zod validation across all API boundaries.

Technical Constraints & Decisions ### 25.1 Architecture Decisions - Queue-driven: BullMQ chosen for Redis backing, built-in retry logic, and TypeScript-native support - Fastify over Express: chosen for performance (2x throughput) and schema-first validation - Prisma ORM: chosen for type-safe database access, migration management, and developer ergonomics - Turborepo: chosen for build caching and parallel task execution in the monorepo - Next.js: chosen for the web UI to leverage SSR, API routes, and the React ecosystem - PostgreSQL RLS: Row-Level Security policies on all tenant-scoped tables as the database-level safety net for multi-tenancy isolation, supplementing application-level Prisma middleware ### 25.2 UI Component Stack (MANDATORY for AI Agents) AI coding agents MUST use the following UI stack for all interface components. This prevents inconsistent styling choices and hallucinated CSS modules: - Styling: Tailwind CSS for all utility-class-based styling. No raw CSS modules, no styled-components, no inline style objects. - Components: shadcn/ui as the component library (built on Radix UI primitives). Use shadcn components for buttons, dialogs, tables, forms, dropdowns, toasts, and navigation. - Icons: Lucide React for all iconography. No mixing icon libraries. - Charts: Recharts for dashboard visualisations (line, bar, area, pie). Consistent across all views. - Forms: React Hook Form with Zod resolvers for all form validation. - Dark mode: Use Tailwind dark: variants with next-themes for theme toggling (US-OPS.04). ### 25.3 Schema Validation (MANDATORY) AI coding agents MUST use Zod for ALL of the following. This is non-negotiable — AI agents are unreliable at handling edge-case nulls without strict schemas: - Fastify API request body validation: Every POST/PATCH endpoint must define a Zod schema. Use fastify-type-provider-zod for Fastify integration. - Fastify API response validation: Response schemas ensure the API contract is enforced even when Prisma returns unexpected shapes. - External API response parsing: EPSS CSV rows, CISA KEV JSON, NVD API responses, PSA/RMM API responses must all be parsed through Zod schemas with .safeParse() to handle malformed data gracefully. - Environment variable validation: Use Zod to validate all env vars at startup (apps/api/src/config.ts). Fail fast if required vars are missing. - Webhook payload validation: Both incoming (from PSA callbacks) and outgoing (to subscriber endpoints) webhook payloads must conform to versioned Zod schemas. ### 25.4 Constraints for Agents - Phase 9 agents MUST be written in Go (cross-compilation, small binary, no runtime dependency) - Go agents MUST be compiled for all six targets: linux/amd64, linux/arm64, darwin/amd64, darwin/arm64, windows/amd64, windows/arm64.
